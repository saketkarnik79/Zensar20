Debugging, Code Review & Refactoring with Copilot
	Copilot does not debug, review, or refactor the code by itself.
	Rather, it helps us think faster, surface risks earlier, and reduce the mechanical effort.
	Use Copilot as:
		A second pair of eyes
		A constraint checker
		A pattern recall engine
		A refactoring process accelerator
	Never use it as:
		An authority
		A decision-maker
		A truth source
	What copilot is good at in debugging?
		Explaining unfamiliar code
		Hypothesizing likely failure modes
		Spotting common async, LINQ, EF, DI mistakes
		Translating stack traces into causes
		Suggesting what to imspect, not what to trust
	What Copilot is not good at?
		Reproducing runtime condition
		Understanding production data
		Knwing system invariants
		Knowing what should happen in your domain
		
	var result = service.GetDataAsync().Result;
	Copilot will most likely flag the following:
		Thread Pool starvation
		Deadlocks in ASP.NET
		Blocking async flow
	Human confirms by;
		Observing request hang
		Checking thread usage
	var result = await service.GetDataAsync();

		Refactoring with Copilot
			When Copilot is excellent for Refactoring:
				Mechanical refactors
				Method extraction
				Naming cleanup
				Async propagation
				LINQ readability improvement
				DTO mapping cleanup
				Boilerplate Reduction
				
			Safe refactoring Rules
				Refactor with constraints or don't refactor at all.
				Allways include:
					Preserve behavior
					No new dependencies
					No logic changes
				Incremental Refactoring
					Anti Pattern
						Refactor this class to be cleaner & more scalable.
					Correct Pattern:
						Rename methods
						Extract methods
						Simplify Conditions
						Stop
					if (x != null && x.Value != null && x.Value.Count > 0)
					Simplify condition for readability.
					Preserve exact behavior.
					
					if (x?.Value?.Any() == true)
			Combine Debugging & Refactoring	
				Debug with Chat
				Confirm Manually
				Refactor with Inline Copilot
				Re-Review with Chat
				Run Tests
			The correct way to ask Copilot to explain the legacy code
				Start with Explaination, Not Critique
					Good Prompt
						Explain what this code does step by step.
						Focus on behavior, dataflow, & side effects.
						Do not suggest improvements yet.
				
					Bad Prompt
						Clean up this legacy mess.

Secure Coding & Compliance with Copilot
	Security & compliance are properties of the system & process and not of the code generator.
	Threat Modeling should come first
		Before writing & generating code, explicitly define:
			Assets:
				PII, PHI, Financial Data
				Credential, Tokens, secrets
				Business-crictical workflows
			Attack Surfaces
				APIs (Endpoints)
				Deserialization & JSON/XML Binding
				Database queries
				Third-Party Integrations
				Background Jobs
			Trust Boundaries
				Client <-> Endpoints
				Endpoints <-> Databases
				Internal Services <-> External Services
	Secure Coding Baselines In .NET
		Input Validation (Never Implicit)
			What not to rely on?
				Model binding alone
				UI Validation
				Client-side checks
			Correct Pattern	
				Explicit servcer-side validation
				Centralized, deterministic rules
				
				public class CreateOrderValidator

							throw new ValidationException("CustomerId is required");

					}
				}

	• Identifying insecure AI-generated code
		AI-generated code is often syntactically correct, idomatic, and reusable - yet insecure.
		Why is it so?
			AI optimizes for plausibility, not the threat models.
			It knows common patterns, not your security posture.
			It favors happy paths.
			It lacks context about regulations, trust boundaries & abuse scenarios.
		The most common red flags
			Authentication
			Authorization
			Cryptography
			Secrets & Key handling
			Payments & Billing
			PII and PHI handling
			Serialization/Deserialization
			File Uploads/Paths
			Dynamic SQL/Raw Queries

Engineering Best Practices & Team Guidelines
	Responsible use of Copilot & AI coding Assistants
	• Guiding Principles
		AI tools accelerate execution; humans own correctness, and accountability.
	• When to trust vs challenge Copilot
		Safe to Trust 
			Boiplerplate & scffolding
			DTOs
			Simple models
			Test Data & mock objects
			Repititive CRUD code
			Configuration wiring
			Logging structure
			YAML pipelines & build steps (after review).
		Copilot output must be challenged line-by-line in the following areas:
			Auth
			Crypto
			Input validation & serialization
			Payment, billing or financial logic
			PII/PHI handling
			Concurrency, async, threading
			EF Core queries affecting large data sets
			Performance-critical or latency-sensitive code
			Compliance-driven code
			Infrastructure, identity & permissions

	• Coding standards enforcement

			All code (human or ai-assisted) must:
				use intention-revealing names
				Keep methods short & single purpose.
				Avoid hidden behavior
				Prefer explicit over clever
				Separate orchestration from decision logic
			Explicit architecture boundaries
